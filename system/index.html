Self-improving agent harnesses as code optimization

← The revise-proxy results — the validated cost work behind §4

The harness is the code we optimize — its skill file, tools, scaffold, memory policy. AlphaEvolve/TTT is the outer search; each rollout is an agent that diagnoses the harness and proposes an edit; the reward is running the improved harness on real tasks and judging the output. For Harvey (a legal agent on Claude Sonnet 4.6), that reward is the expensive part — the focus of this system.

A harness turns a base model into a working agent: a system prompt / skill file, a tool set, a control-flow scaffold, a memory policy, and self-check logic. Seen this way, improving a harness is a code-optimization problem — an editable artifact plus a numeric reward — so AlphaEvolve (evolve an archive of variants) and TTT (test-time training — the same loop, but also RL-trains the proposer) apply directly. The one twist: a good edit can't be guessed blind, so the proposer is an agent that first looks at why the harness fails.

The system at a glance

Two nested loops. Outer: AlphaEvolve/TTT samples a parent harness from the archive and files back scored children. Inner (per NanoRollout): an agentic proposer inspects traces and edits the harness. Reward: run the improved harness on the calibration set and judge the deliverables. Solid = exists today; dashed = being built.

Reading the diagram — one turn of the loop

  • Sample a parent. The archive (MAP-Elites + islands, today's OpenEvolve ProgramDatabase) hands a promising harness variant to one NanoRollout. This whole rollout is one task=“evolve” episode — one mutation of the outer search.
  • Propose an edit, agentically. The new part: instead of one model call emitting a diff blind, an agent with tools reads the harness's execution traces and failing transcripts, identifies the inefficiency, tries a variant in a sandbox, and writes the diff. One proposal = one agent trajectory, grounded in observed failures. First we edit the skill file; tools, scaffold, and memory come later.
  • Score it — the expensive part. Harness-eval runs the improved harness over the calibration set (N tasks × k repeats), the deliverables are graded by the judge, and the mean pass-rate is the reward R̂. With Sonnet as both the harness endpoint and the judge, this dominates cost.
  • File it back. The scored child returns to the archive; winners survive and get built on. In TTT mode the same rollout is also used to RL-train the proposer agent.

The harness decomposes into editable regions; the optimizer may touch any subset. We start with the highest-leverage, lowest-risk one and widen later.

The Harvey harness is a 6-tool loop (bash / read / write / edit / glob / grep). The skill file is the cheapest, highest-variance surface, which is why it goes first.

2 · The twist: an agentic inner loop

Was: a single model call

Hand a frozen model the parent harness; it emits a diff in one shot. Cheap and low-variance, but it proposes blind — it never sees why the current harness fails. Most such edits are shots in the dark.

Now: an agent with tools

“Propose a better harness” is itself a task: read the code, run it on a few tasks, read the failing transcripts, try a variant, then write the edit. The proposal is a diagnosis, not a guess — one agent trajectory instead of one token sequence.

The optimizer and the optimized are the same kind of object — an agent harness improving an agent harness. That symmetry is the door to self-improvement: eventually the proposer can improve its own harness. The open empirical question (to measure first): does the richer agentic proposal beat the cheap single call per unit of compute?

3 · Why the reward is the hard part

We can never measure a harness's true quality — its success rate over the full task distribution D the customer cares about — because we never have D (it's unknown, drifts, and the long tail is the part that matters). Every reward is a proxy: mean pass-rate over N sampled tasks × k repeats, with two error sources — sampling (which/how many tasks, fixed by N and subset design) and stochastic (run-to-run variance, fixed by k). And one evaluation costs N × k × (per-task agent cost), where the per-task cost is large. So the reward is opaque (we don't know the target) and rationed (we can afford few runs). The evaluation set is therefore not sampled — it is engineered.

4 · Three ways we make the reward cheap

This is exactly what the two prior projects attacked, plus the fidelity machinery the design calls for. The goal is the same throughout: cut the dollars per scored candidate without distorting which harness ranks best. Each attacks a different factor of the cost N × k × (agent + judge) — the revise proxy cuts the agent-rollout cost, the ensemble judge cuts the grading cost, and the cascade cuts how many runs you pay for (the ①②③ below match the diagram).

① Revise proxy validated

Predict whether a skill helps without re-running the agent: take the deliverable Sonnet already produced without the skill and have it revise that draft to follow the skill, in one call; grade the revision.

Tracks the real full-run lift at in-set r = 0.76, at ~3.6× lower cost ($0.53 vs the $1.90 full Sonnet run). A cheap reviser (Haiku) is too weak (r=0.27).

② Judge-cost opt validated

The judge is an ensemble: gpt-5-mini grades all criteria, gpt-5 gives a second opinion, and Sonnet is the tie-breaker only on disagreement — about 1.86% of criteria.

0.16% false-pass; Sonnet judges only 1.86% of criteria (vs 100% per-criterion); Batch API another −50%. Grading stops being the bottleneck.

③ Fidelity cascade being built

Score candidates cheaply first and only promote survivors: smoke→proxy→medium→full (e.g. 20→50→100 calibration tasks), with repeats k raised only for near-final candidates.

Plus eval-set engineering: stratify by area, keep high-discrimination tasks, synthesize the missing tail, rotate to resist overfitting.

Validated numbers come from the calibration study on 45 Harvey tasks × 3 skills (30 in-set + 15 held-out). Full results and method: the revise-proxy results site.

One outer proposal spawns N×k inner rollouts, so a batch of M candidates is M×N×k concurrent agent rollouts — the cost (and the scheduling) is dominated by the inner task rollouts, not the proposer. That is why §3–4 — making the reward cheap — is the whole game.
