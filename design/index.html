Agentic AlphaEvolve: a research agent that improves the Harvey agent's harness

We optimize the Harvey harness with an evolutionary loop whose mutator is a code-writing research agent — a different agent from the Harvey agent whose harness it edits, with no gradient step in the current loop. The cornerstones:

  • Filesystem = the state. The whole workspace is one OpenEvolve Program the archive stores, mutates & selects on: harness/ (the Harvey harness under evolution) · CLAUDE.md (read-only human prior, locked so it can't drift) · rollouts/ (the standard eval's trajectories) · scratch/ (ephemeral, dropped each rollout) · and .claude/ — since the mutator runs on a Claude Code harness, its own files sit here: skills/ (skills & workflows it writes), memory/ (its verified lessons), tools/ (its eval proxy + strong-model APIs).
  • Two distinct agents. A frozen Harvey agent (weights never move) does the legal work; a separate research agent — the mutator — edits the harness around it.
  • Code can change any part of the harness. The mutator writes code, so it can edit anything: skill → tools → control-flow scaffold (e.g. compaction) → eventually the Harvey agent's weights.
  • The mutator works in its own sandbox + filesystem workspace (shell + code). A tool for the Harvey agent to use → edits harness/; a tool that helps the mutator optimize (e.g. the eval proxy, seeded with our pre-built one) → .claude/tools/, which it can add to & improve.
  • Two evals, strictly separated. The mutator's private proxy evals (run mid-rollout to triage its own trial edits) vs. the canonical standard eval at the end of each rollout — the only score OpenEvolve selects on, and itself any common procedure we standardize on — here, the real harness run graded by a fixed ensemble judge, fail-fast cascaded (§4).

The goal is to optimize the Harvey harness — the skill, tools, and control-flow scaffold wrapped around the Harvey agent. The design is two coupled loops. The inner loop is the Harvey agent doing legal work: it runs the current harness on calibration tasks and emits rollouts (a trace + a deliverable), weights frozen. The outer loop is a different agent — the research agent (the mutator) — that reads those rollouts and writes code to modify the Harvey harness: new skills, tools, scaffolding. Many cheap inner rollouts (one pass over the task set) feed one expensive outer edit (a multi-step agent trajectory); repeat.

In one line: this is AlphaEvolve (DeepMind) — evolutionary search that keeps a population of programs, scores each, and mutates the winners — with two changes: (1) the program being evolved is the Harvey harness, and (2) — the idea that matters — the mutator isn't a single blind LLM diff but an agent that investigates: it reads the failing rollouts, diagnoses the cause, and writes the code. Everything else (archive, score, evolve the winners) is standard AlphaEvolve.

The whole system in one picture

Four stations, one cycle ①→④. ① the archive samples a parent harness (a filesystem state) → ② the research agent (mutator), in its sandbox + workspace, reads the dumped rollouts and writes code to edit the harness → ③ the improved harness runs on the Harvey LAB dataset → rollouts (trace + deliverable) → ④ a standard eval scores the deliverables into combined_score, and OpenEvolve files the (state, score) back into the archive and picks the next parent. The standard eval's trajectories are carried into rollouts/ for the next round (the mutator's optional proxy-eval runs go to scratch/ instead). Solid = exists/merged, dashed = PR-open or to-build.

Reading the diagram — one turn of the loop

  • Sample a parent harness. The archive — a population of harness variants (each a filesystem state) kept in a diversity grid (MAP-Elites + islands: keep the best variant in each niche, not just one global best) — hands a promising harness to the outer loop.
  • Edit the harness, agentically. The research agent (mutator, GLM-5.2 or Claude), warm-started from its CLAUDE.md and reading the parent's dumped rollouts, diagnoses what's missing and writes code to edit the harness — a skill, a tool, scaffolding. This is the one change from vanilla AlphaEvolve: an agent that investigates in its own sandbox, not a one-shot blind diff. It records what it learns in .claude/memory/. The result is an improved harness.
  • Run it on the Harvey LAB dataset. The inner loop is simple: take the improved harness and run it over the task set — one pass, the Harvey agent's weights frozen (0th-order) — producing a rollout (trace + deliverable) per task. No per-rollout sandbox. The rollouts are dumped into the mutator's rollouts/ for the next round.
  • Score it — the standard eval. The deliverables are graded into one combined_score — success plus cost/steps (§3). This is the canonical number OpenEvolve selects on (distinct from the mutator's own optional proxy evals). OpenEvolve files the (state, score) into the archive and picks the next parent; the population improves over rounds.

1 · What we optimize: the harness, as code

The harness decomposes into editable regions. The mutator edits them as code, so in principle nothing is off-limits — the ladder below is really about how much power we hand it, lowest-risk first. We start with the skill file (cheapest, highest-variance), then tools, then the control-flow scaffold (e.g. the LAB blog's finding that compaction helps — the mutator can add compaction to the scaffold as code), and eventually the Harvey agent's weights. Which levers are worth trying first is itself seeded from human knowledge into CLAUDE.md.

Seen this way, improving the harness is a code-optimization problem — an editable artifact plus a numeric reward — exactly what AlphaEvolve needs. Concretely the artifact is the filesystem workspace — harness/ + CLAUDE.md + rollouts/ + the mutator's .claude/ (skills, memory, tools), minus the ephemeral scratch/. That directory is one OpenEvolve Program, and it's what the archive stores, mutates, and builds winners on.

2 · The outer loop — an agentic, warm-started proposer

This is the new idea, so it gets the most attention. In vanilla AlphaEvolve a frozen model emits one diff, blind. Here, proposing a better harness is itself an agent task — we call this agent the mutator (a.k.a. the research agent or proposer): three names for one agent throughout this doc.

Why agentic, not a single call

A good edit can't be guessed blind. It requires looking: read the failing traces, see which tool the Harvey agent reached for and botched, try a variant in the shell. So one proposal becomes one agent trajectory — a diagnosis grounded in observed failure, not a shot in the dark.

Why warm-started

The proposer doesn't start from zero. We seed it on its filesystem: human prior into CLAUDE.md (Harvey's LAB blogs — emit an artifact, cite spans, validate→revise — plus the A+ playbook, the GLM-5.2 endpoint, what predicts a skill's lift), our pre-built proxy and API calls to strong closed models into .claude/tools/, and a baseline harness/ that already includes an initial compaction we built. So it starts warm — informed edits from turn one — without parroting any of it (a seeded idea still has to beat the parent on held-out tasks to survive).

The mutator's workspace: one filesystem, structured

The mutator isn't handed a digest — it's handed a sandbox (a filesystem, PR #86, and a local shell, PR #87) and writes code in it like a research engineer. That sandbox belongs to the outer loop — the inner loop (§3) has none. The filesystem holds four shared parts:

  • harness/ — the Harvey harness under evolution: the skill file, the control-flow scaffold, the agent loop, and the tools the Harvey agent calls (citation-checker, compaction, …). When the mutator builds a tool for the harness or adds compaction, it edits code here. This is what the inner loop runs.
  • CLAUDE.md — the human prior, auto-loaded by Claude Code (above): the validated A+ playbook, our findings (GLM-5.2 endpoint, what predicts a skill's lift), a starter menu of scaffold-level levers (compaction, planner, retrieval, memory), and links to the Harvey LAB blogs. Locked read-only so the prior stays faithful across iterations — the mutator reads it but can't let it drift.
  • rollouts/ — the standard eval's trajectories (the canonical run on the dataset, §4): every trajectory (trace · deliverable · score), dumped here each round and carried over. The mutator's optional proxy-eval runs do not land here — those go to scratch/. Raw evidence, no taxonomy pre-digesting it (a deliberate reversal of feedback.py) — the raw dump lets the mutator notice failure modes a fixed taxonomy's authors would have missed.
  • scratch/ — the mutator's ephemeral workbench: provisional hypotheses, half-written tools, and the artifacts from its optional proxy evals (which trigger the inner loop too, but land here, not in rollouts/). Not saved — discarded at the end of the rollout; only what graduates to harness/ or .claude/ persists.

And because the mutator runs on a Claude Code harness, its own self-authored files live under .claude/ in that harness's native layout:

  • .claude/skills/ — skills & workflows the mutator writes for itself: reusable procedures it discovers (“how to diagnose an over-retrieval failure”, “the steps to add and validate a harness tool”). Its own playbook — distinct from the Harvey agent's skill in harness/ — accumulating so the next mutator works smarter.
  • .claude/memory/ — only verified lessons, promoted out of scratch/ once they survive a held-out check (“the citation tool cut errors but added 3 steps”). The durable memory the next mutator reads back.
  • .claude/tools/ — the mutator's OWN tools: tools that help it optimize the harness, separate from the harness itself. Seeded from the start with the agent/judge proxy it can launch mid-rollout (under a cost cap) to triage edits it just proposed, and API calls to strong closed models (GPT-5, Claude, Opus) for hard reasoning. It can also write even cheaper evals here (e.g. a smoke test on a handful of examples). In v1 we may keep .claude/tools/ read-only — the mutator uses the seeded tools but doesn't yet build its own, deferring “improve your own toolkit” to a later opt-in. The Harvey agent never sees any of these.

The mutator works by writing code: it edits the harness (tools, skills, scaffold) and records what it learns in .claude/memory/ — each lesson starting in scratch/ as a hypothesis, promoted to .claude/memory/ only after a held-out check (grounded in eval counts, not two cherry-picked traces). Because the proxy already sits in .claude/tools/, the mutator can also launch it mid-rollout — under a cost cap — to triage the edits it's trying out before committing one. Going further is the optional, later step: with .claude/tools/ writable (not v1's read-only default), it can build its own sharper evals — fork the proxy into a cheaper stand-in or judge, or write a smoke test over a handful of examples — drafting them in scratch/ first. Either way these proxy evals stay the mutator's private instrument: it sharpens what it uses to decide, but it cannot touch the canonical standard eval OpenEvolve selects on (§4) — or it would be grading its own homework.

The research agent and the Harvey agent are two distinct agents — one does legal work, the other writes code to modify the Harvey agent's harness. (They happen to be the same kind of object, an agent harness, so the same machinery could later be turned on the research agent's harness too — the TTT direction in §6, not today.) The open empirical question to measure first: does the richer, pricier agentic proposer beat a cheap single call per unit of compute? The proposer is the only new cost we add; every §4 win reduces the other (eval) cost — so whether the proposer earns its added cost is the gating experiment.

3 · The inner loop — 0th-order, no gradients

The inner agent is “trained” without touching its weights. We don't RL-finetune GLM-5.2; we change the harness around it — the tools it's handed, the skill in its prompt, what its memory carries. That is the 0th-order move: the research agent changes the inner agent's behavior through the harness, not through gradients. The 0th-order move needs no training infrastructure, so the inner loop ships first, as the cheaper half of the system. For now — the far end of the ladder (§1) is the mutator writing training code that updates these weights, at which point the inner loop leaves 0th-order. Note that's a different escalation from TTT on the proposer (§6): this trains the agent being optimized; TTT trains the optimizer.

Crucially, we push it toward using tools more efficiently, not just scoring higher — Harvey's own data shows the trained agent cut reads 44→24. So the reward is quality and cost (steps / tokens), and efficiency also becomes an archive axis (§6) — otherwise a tool that doubles pass-rate by tripling tool calls would look like progress.

The inner loop is deliberately simple: take the improved harness and run it on the Harvey LAB dataset (our calibration set) — one pass over the task set — producing a rollout (trace + deliverable) per task. There is no per-rollout sandbox or special machinery here; the shell, code, and filesystem all live in the outer loop (the mutator's sandbox — PR #86 + PR #87, §2), not here. This canonical run is the standard eval's (§4); its trajectories are dumped into the outer loop's rollouts/ and carried over for the mutator to read next round. (The mutator's optional proxy evals also run the inner loop, but their artifacts land in scratch/ — ephemeral, not carried.) Today it's a single pass per task; a within-task draft→check→revise loop is a later option.

4 · The reward — and why its cost gates the search

Two evals run here, and keeping them apart matters. During a NanoRollout the mutator launches its own cheap proxy evals (§2) to triage the candidate edits it is trying out — private, cost-capped, just to decide what to commit. Those trial edits never reach selection on their own. At the end of every NanoRollout, a single standard eval runs on the committed harness → one combined_score, and that is what OpenEvolve files into the archive and selects on. The standard eval is whatever common procedure we standardize on, the one hard requirement being that it's the same for every candidate so scores compare. Our choice: run the real harness (not a cheap agent stand-in) over the task set and grade with a fixed batched-ensemble judge, fail-fast cascaded so most candidates die cheap. We deliberately keep no agent proxy in this canonical path — a proxy calibrated on one harness mis-ranks the moment the mutator changes the harness, and the mutator could learn to game it; cheap agent proxies stay the mutator's private triage (§2). Harvey's own LAB work lands the same way: they run real rollouts of every harness variant against a fixed grader (GPT-5 Mini, validated to >97% agreement with a GPT-5.5 / Opus-4.8 frontier consensus at 0.99 Pearson), batching rubric criteria per call for a 40–100× cost cut — their cheap-eval work is all judge-side, never an agent proxy. (So the proxy isn't redundant with the standard eval: the proxy triages the mutator's own in-flight ideas; the standard eval scores the one harness it commits.) The rest of this section is about making that end-of-rollout eval cheap, because it runs every step.

Scoring one candidate means running the harness on N calibration tasks and judging every deliverable — the agent rollouts dominate the bill (the batched judge is a rounding error beside them). The agent is already cheap — GLM-5.2 at ~$1/task, about half of Sonnet — so the main lever is the fail-fast cascade: a $0 structural gate first (did the harness even run and emit a non-empty, well-formed deliverable? — catches a mutator that shipped broken code, before any paid rollout), then a smoke run on a small task subset, then the fuller set, so most candidates die cheap. Two more levers ride alongside it: engineered eval sets (stratify / discriminate / rotate, so budget is spent only where candidates actually separate) and an escalate-on-disagreement judge (the cheap ensemble grades everything; a stronger judge breaks only the ties). The cascade is what lets one generation afford many candidates.

Mind the noise floor. The reward is stochastic — a strong harness can swing several points run-to-run on a small eval. Evolving on a signal below that floor just promotes lucky harnesses. So a candidate has to beat its parent by more than the measured run-to-run σ (more seeds / tasks, not one pass) before it's archived, and the cascade's rungs are sized by variance, not only by cost.

Keeping the proposer honest. The proposer's job is to maximize a judge — a reward-hacking risk. The simplest, cleanest guard, and all we need for now: evaluate on a held-out set. A harness that only looks good on the tasks the search optimized over won't hold up on tasks it never saw, so the held-out score is the honest verdict. (Richer anti-gaming gates — judge rotation, grounding checks, a canary set — can come later if they prove necessary.)

5 · A later direction: ToolOrchestra not in v1

A wishful extension we've wanted from the start — flagged here as a separate, post-v1 direction. ToolOrchestra trains a small orchestrator model (8B) to coordinate calls to bigger models and specialized tools, with RL on outcome-, efficiency-, and preference-aware rewards. Instead of one big model running every step, a cheap orchestrator decides per step whether to answer itself or call out — paying for a big model only when a step needs it. The payoff is steep: their 8B orchestrator beats GPT-5 on Humanity's Last Exam (37.1% vs 35.1%) at 2.5× the efficiency, and surpasses GPT-5 on tau2-Bench / FRAMES at ~30% of the cost.

One model everywhere → a cheap orchestrator over a toolbox. Big-model quality on the hard steps, a fraction of the cost on the easy ones.

Two places this drops into the present design — in both, “the model endpoint” simply becomes a richer object:

① The mutator becomes the orchestrator

The big API models already sit in the mutator's .claude/tools/ from day one (§2). The post-v1 step is to make the mutator itself a small model, TTT-trained (the repo's namesake) to orchestrate them — learning to call the expensive models sparingly. This directly answers §2's open question — does the agentic proposer beat a cheap single call per unit of compute? — ToolOrchestra makes the proposer cheap by construction.

② The Harvey endpoint becomes an orchestra

Today the harness endpoint is one model (GLM-5.2). Because the endpoint lives in harness/, the research agent can rewrite it into a ToolOrchestra — a small orchestrator that routes each step to the right model or legal tool (retrieve, cite-check, draft). Big-model quality on the hard steps, a fraction of the cost elsewhere; just another rung on the edit ladder (§1).

Why it's a clean fit: our reward is already quality and cost (§3–4) — exactly ToolOrchestra's multi-objective signal; the archive already selects on combined_score; and TTT on the proposer (§6) is the very training loop it needs. But it's explicitly out of scope for v1 — it lands only once the base loop works.

6 · The hard parts (design choices & open questions)

The two most novel mechanisms — the agentic proposer and self-generated tools — are also the least built. Here is where the real design decisions live.

What can the mutator change? (the edit ladder)
Because every edit is code, the editable surface widens as we grant more power: skill (prose) → tools (new actions) → scaffold (compaction, planner, validate→revise) → eventually training code that updates the Harvey agent's weights. We climb it lowest-risk-first; each rung needs stronger sandboxing and validation than the last, and the top rung (touching weights) stays gated behind the cheaper rungs actually paying off. The same power could later let it improve its own instruments (an optional add-on, §2): a cheaper agent stand-in or sharper judge proxy for its private eval — never the canonical standard eval it's scored by. That code-can-do-anything power is why the sandbox and budget guards (below) are load-bearing, not nice-to-have.
What's writable, and what's locked?
The mutator has shell access to the whole workspace and is steered by its prompt toward the right edits — the locks below are deliberate infra, not technical limits. CLAUDE.md is read-only so the seeded human prior stays faithful across iterations (the mutator can't let it drift). In v1 we may also lock .claude/tools/ read-only — the mutator uses the seeded proxy + model-APIs but doesn't yet build its own, deferring “improve your own toolkit” to a later opt-in. Everything else (harness/, scratch/, .claude/memory/, .claude/skills/) it writes freely.
What is a “new tool,” and how is it born?
Concretely: the proposer writes a small Python function with a typed signature (e.g. a citation-checker that verifies every cited span exists in the source). It must pass a validation rung before it can score — does it run, does it improve a held-out micro-task (including adversarial cases where the right answer is “fail”, so a tool that always returns true is caught) — all inside the mutator's sandbox (§2), isolated from the host. Once it passes, it graduates from scratch/ into the harness (harness/) and the Harvey agent can call it. A “skill” (prose) edits the prompt; a “tool” (code) extends the Harvey agent's action space — both are edits to harness/. (A proxy the mutator builds for its own evals is a different thing: that lives in .claude/tools/, §4.) We start with skills (lower risk) and graduate to tools.
How do tools live in the archive?
A harness variant's genotype is its persisted filesystem state — harness/ + rollouts/ + the read-only CLAUDE.md + the mutator's .claude/ (skills, memory, tools); the ephemeral scratch/ is dropped. Tools are namespaced and content-hashed so children inherit them without collision, and a tool-bloat guard (a cost axis + periodic consolidation) retires redundant harness tools — otherwise “build on winners” accretes cruft and inflates the Harvey agent's prompt until adding more tools lowers the score.
What are the MAP-Elites axes?
MAP-Elites is only as good as its behavior axes. The most natural one here is which harness component the variant edited — skill, tools, or scaffold (the edit-ladder rungs of §1) — so the grid keeps a diverse population working on each region, instead of the whole population concentrating on the cheap skill edits. Alongside it: resource axes (skill-file length, number of tools, steps/task, cost/task — the efficiency signals from §3), and at least one capability axis so the grid preserves variety in what a harness is good at, not just how expensive it is — e.g. a per-task-cluster performance bucket (drafting vs. retrieval vs. cite-checking). The “where does it fail” axis comes from the failure clusters visible in the rollouts, not the feedback.py taxonomy we're replacing. Together these keep the search from collapsing to a global top-k.
Search vs. evaluation budget.
One outer proposal fans out to N×k inner rollouts; the agentic proposer is itself now multi-step and pricey. The standard-eval cascade promotes a candidate smoke→subset→full so most die cheap, and the proposer's per-edit cost is tracked against the single-call baseline — that ratio is the core experiment (§2).
Cost monitor, cap & failure — the mutator spends real money.
Once the mutator can launch its own experiments (proxy evals, strong-model API calls), it's an unbounded spender, so the outer loop needs a hard budget guard. Two-level cap: a per-NanoRollout ceiling and a whole-run ceiling, so no single lineage drains the search. How it's enforced — a synchronous meter, not a background watcher. Cost only moves when the agent acts, so there's nothing to poll between actions; the orchestrator keeps a running tally (from the token counts it already has for billing) and checks it at three call-sites: before each rollout starts (the per-run cap), after every mutator turn (catches pure-reasoning runaway — granularity is one turn, so worst-case overage is a single turn), and before every paid tool call — the proxy-eval and strong-model-API tools in .claude/tools/, not the (free) shell & file edits. Because that last gate is checked pre-spend, the big discrete costs incur zero overage. (A background timer would be both wasteful — nothing changes between actions — and unsafe: it leaves a race window to fire a $30 eval just after a poll said “fine.”) When the tally crosses the cap the next metered action throws instead of running, and that exception is the “out of budget, commit now” signal. Three outcomes, kept distinct: (i) hit the cap but produced a valid edit → score it normally (the cap is a ceiling, not a failure); (ii) crashed / looped / no valid edit → floor score, prune the lineage, don't block the loop; (iii) its own experiment errored → that's a scratch/ problem and must not poison the standard eval. A per-source cost ledger (mutator reasoning / proxy evals / strong-APIs) lands in scratch/. Soft + hard: remaining() is exposed to the mutator so it self-paces (“$8 left → one smoke test, not a full sweep”) — in the common case the cap never trips and the throw is just the guardrail. Finally we turn a real kill into a learning signal: CLAUDE.md encourages the mutator, on a budget-kill, to write a short lesson to .claude/memory/ (“that proxy config blew the cap”) — which survives compaction and carries to the next iteration, so the run gets cheaper over time instead of repeating the same expensive mistake.
Won't dumping everything blow the mutator's context?
The dump is large — the N×k traces plus the full harness. The mutator doesn't read it all into context; it gets a filesystem + shell and navigates it (grep for the failures, diff the harness against its parent, sample a few traces, re-run one case), the way an engineer triages a log directory. So “dump everything” is cheap to produce (just write the inner loop's artifacts to disk) and the mutator pays only for the slice it chooses to read.
The mutator's own context — compaction & tool set.
Even navigating rather than reading everything (above), the mutator's own trajectory gets long. It runs on a Claude Code harness, which auto-compacts — past a token threshold it summarizes the earlier turns and continues. That's the same mechanism Harvey's harness uses for the inner agent (send the transcript back to the same weights under a summarization prompt; in their data ~10% of base rollouts hit the context limit, and compaction cut tool calls 104→42 and payload tokens 461k→250k) — so compaction shows up at both layers: a tunable scaffold lever inside harness/ for the inner agent (§1), and a free property of the Claude Code harness for the outer mutator. The catch is that auto-compaction is lossy, so nothing load-bearing can live only in the summary — which is exactly why durable state goes to the filesystem (.claude/memory/ for verified lessons, scratch/ for working notes): “filesystem = state” makes the mutator robust to its own compaction. Tool set: Claude Code hands it Bash + file ops + search + subagents + web by default; for the outer loop we restrict it to shell + file + search + the metered .claude/tools/ (+ subagents for parallel triage) and drop web (non-reproducible, a wander/exfil risk) unless we deliberately want it for warm-start research — with the expensive tools gated behind the budget cap above.
Evolution now, TTT later.
Today the mutator is frozen (pure evolution — improvement accumulates as scored harness variants in the archive and verified findings in .claude/memory/). The speculative next step, and the repo's namesake, is test-time training: RL-train the proposer so it specializes at proposing harness edits (distinct from the §1 rung where the mutator trains the inner agent — here we train the proposer). It isn't free — training it on the same reward the archive is selected on couples two optimizers to one signal (compounding the reward-hacking risk above), and the target is non-stationary (the harness it edits keeps changing). So TTT likely needs a held-out reward channel distinct from the search judge, or it just amplifies whatever that judge is blind to. Same loop, consumed twice — carefully.

8 · Where we are, and the gap

Today (the harvey-evolve-prototype branch): a complete, well-engineered single-loop optimizer. OpenEvolve mutates the skill-file prose, injects it via HARVEY_PLAYBOOK, runs the Harvey harness, scores with its standard eval (today, the ensemble judge), files variants into MAP-Elites, and feeds a heuristic failure-mode taxonomy (its feedback.py) back into the next mutation. The reward and the cost discipline are production-quality. But the “proposer” is one LLM rewrite, the feedback is a fixed heuristic, the artifact is a single markdown file, and there's no inner agent.

Landing now: the mutator's sandbox — PR #86 (filesystem) and PR #87 (local shell), the outer loop's shell + code — plus PR #88 wiring OpenEvolve to NanoRollout (on a sibling branch). These turn the single prose-rewrite into a real agentic loop.

The gap to close (this design):

  • A real inner loop — a frozen, 0th-order Harvey agent running the improved harness on the dataset to produce rollouts (a within-task draft→check→revise loop is a later option), not one prose-rewrite per outer step.
  • A genuinely agentic proposer (warm-started + tool-using), not just a seeded artifact mutated by a bare LLM.
  • Raw-dump feedback, not a taxonomy. Replace feedback.py with dumping the inner loop's whole evidence package (all trajectories + the harness code) to the filesystem, and let the agentic mutator extract its own feedback and propose the next tools & skills (§2).
  • A verify → new-tools path (§6) that grows the action space — synthesize, validate, and register tools — rather than only rephrasing a playbook, kept honest by anti-gaming gates.

The target end state: a research agent that writes the Harvey agent's tools, validates them on cheap rollouts, and keeps the ones that survive on held-out tasks — optimizing the Harvey harness automatically, with no gradient step in the loop.
