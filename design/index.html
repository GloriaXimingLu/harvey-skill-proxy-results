Agentic AlphaEvolve: a research agent that improves the Harvey agent's harness

We optimize the Harvey harness with an evolutionary loop whose mutator is a code-writing research agent — a different agent from the Harvey agent whose harness it edits, with no gradient step in the current loop. The cornerstones:

  • Filesystem = the state. The whole workspace is one OpenEvolve Program the archive stores, mutates & selects on: harness/ (the Harvey harness under evolution) · tools/ (tools that help the mutator optimize — e.g. the eval proxy; seeded with our pre-built one, the agent adds & improves more) · knowledge/ (read-only human prior — blogs, our prior attempts) · rollouts/ (full trajectories dumped from the inner loop) · lessons/ (the agent's own written knowledge & memory). The working scratch/ is ephemeral — dropped at the end of each rollout.
  • Two distinct agents. A frozen Harvey agent (weights never move) does the legal work; a separate research agent — the mutator — edits the harness around it.
  • Code can change any part of the harness. The mutator writes code, so it can edit anything: skill → tools → control-flow scaffold (e.g. compaction) → eventually the Harvey agent's weights.
  • The mutator works in its own sandbox + filesystem workspace (shell + code). A tool for the Harvey agent to use → edits harness/; a tool that helps the mutator optimize (e.g. the eval proxy, seeded with our pre-built one) → tools/, which it can add to & improve.
  • Two evals, strictly separated. The mutator's private proxy evals (to decide what to propose) vs. the canonical standard eval at the end of each rollout — the only score OpenEvolve selects on.

The goal is to optimize the Harvey harness — the skill, tools, and control-flow scaffold wrapped around the Harvey agent. The design is two coupled loops. The inner loop is the Harvey agent doing legal work: it runs the current harness on calibration tasks and emits rollouts (a trace + a deliverable), weights frozen. The outer loop is a different agent — the research agent (the mutator) — that reads those rollouts and writes code to modify the Harvey harness: new skills, tools, scaffolding. Many cheap inner rollouts (one pass over the task set) feed one expensive outer edit (a multi-step agent trajectory); repeat.

In one line: this is AlphaEvolve (DeepMind) — evolutionary search that keeps a population of programs, scores each, and mutates the winners — with two changes: (1) the program being evolved is the Harvey harness, and (2) — the idea that matters — the mutator isn't a single blind LLM diff but an agent that investigates: it reads the failing rollouts, diagnoses the cause, and writes the code. Everything else (archive, score, evolve the winners) is standard AlphaEvolve.

The whole system in one picture

Four stations, one cycle ①→④. ① the archive samples a parent harness (a filesystem state) → ② the research agent (mutator), in its sandbox + workspace, reads the dumped rollouts and writes code to edit the harness → ③ the improved harness runs on the Harvey LAB dataset → rollouts (trace + deliverable) → ④ a standard eval scores the deliverables into combined_score, and OpenEvolve files the (state, score) back into the archive and picks the next parent. The standard eval's trajectories are carried into rollouts/ for the next round (the mutator's optional proxy-eval runs go to scratch/ instead). Solid = exists/merged, dashed = PR-open or to-build.

Reading the diagram — one turn of the loop

  • Sample a parent harness. The archive — a population of harness variants (each a filesystem state) kept in a diversity grid (MAP-Elites + islands: keep the best variant in each niche, not just one global best) — hands a promising harness to the outer loop.
  • Edit the harness, agentically. The research agent (mutator, GLM-5.2 or Claude), warm-started from its knowledge/ and reading the parent's dumped rollouts, diagnoses what's missing and writes code to edit the harness — a skill, a tool, scaffolding. This is the one change from vanilla AlphaEvolve: an agent that investigates in its own sandbox, not a one-shot blind diff. It records what it learns in lessons/. The result is an improved harness.
  • Run it on the Harvey LAB dataset. The inner loop is simple: take the improved harness and run it over the task set — one pass, the Harvey agent's weights frozen (0th-order) — producing a rollout (trace + deliverable) per task. No per-rollout sandbox. The rollouts are dumped into the mutator's rollouts/ for the next round.
  • Score it — the standard eval. The deliverables are graded into one combined_score — success plus cost/steps (§3). This is the canonical number OpenEvolve selects on (distinct from the mutator's own optional proxy evals). OpenEvolve files the (state, score) into the archive and picks the next parent; the population improves over rounds.

1 · What we optimize: the harness, as code

The harness decomposes into editable regions. The mutator edits them as code, so in principle nothing is off-limits — the ladder below is really about how much power we hand it, lowest-risk first. We start with the skill file (cheapest, highest-variance), then tools, then the control-flow scaffold (e.g. the LAB blog's finding that compaction helps — the mutator can just write it into the loop), and eventually the Harvey agent's weights. Which levers are worth trying first is itself seeded from human knowledge into knowledge/.

Seen this way, improving the harness is a code-optimization problem — an editable artifact plus a numeric reward — exactly what AlphaEvolve needs. Concretely the artifact is the filesystem workspace — harness/ + tools/ + the seeded knowledge/ + the dumped rollouts/ + the carried lessons/ (the ephemeral scratch/ is dropped at the end of each rollout). That directory is one OpenEvolve Program, and it's what the archive stores, mutates, and builds winners on.

2 · The outer loop — an agentic, warm-started proposer

This is the new idea, so it gets the most attention. In vanilla AlphaEvolve a frozen model emits one diff, blind. Here, proposing a better harness is itself an agent task — we call this agent the mutator (a.k.a. the research agent or proposer): three names for one agent throughout this doc.

Why agentic, not a single call

A good edit can't be guessed blind. It requires looking: read the failing traces, see which tool the agent reached for and botched, try a variant in the shell. So one proposal becomes one agent trajectory — a diagnosis grounded in observed failure, not a shot in the dark.

Why warm-started

The proposer doesn't start from zero. We seed it on its filesystem: human prior into knowledge/ (Harvey's LAB blogs — emit an artifact, cite spans, validate→revise — plus the A+ playbook, the GLM-5.2 endpoint, what predicts a skill's lift), our pre-built proxy into tools/, and a baseline harness/ that already includes an initial compaction we built. So it starts warm — informed edits from turn one — without parroting any of it (a seeded idea still has to beat the parent on held-out tasks to survive).

The mutator's workspace: one filesystem, structured

The mutator isn't handed a digest — it's handed a sandbox (a filesystem, PR #86, and a local shell, PR #87) and writes code in it like a research engineer. That sandbox belongs to the outer loop — the inner loop (§3) has none. The filesystem is organized into subfolders — one seeded and read-only, the rest written as the loop runs:

  • harness/ — the Harvey harness under evolution: the skill file, the control-flow scaffold, the agent loop, and the tools the Harvey agent calls (citation-checker, compaction, …). When the mutator builds a tool for the harness or adds compaction, it edits code here. This is what the inner loop runs.
  • tools/ — the mutator's OWN tools: tools that help it optimize the harness, separate from the harness itself — e.g. the agent/judge proxy it uses for cheap evals (§4). Seeded with our pre-built proxy so it starts warm; the mutator adds & improves more here (graduated from scratch/). The Harvey agent never sees these.
  • knowledge/ — human prior, seeded up front & read-only (above): the Harvey blogs, our findings (A+ playbook, GLM-5.2 endpoint, what predicts a skill's lift), and a starter menu of scaffold-level mechanisms (compaction, planner, retrieval, memory). The mutator reads from here but never writes to it.
  • rollouts/ — the standard eval's trajectories (the canonical run on the dataset, §4): every trajectory (trace · deliverable · score), dumped here each round and carried over. The mutator's optional proxy-eval runs do not land here — those go to scratch/. Raw evidence, no taxonomy pre-digesting it (a deliberate reversal of feedback.py) — the raw dump lets the mutator notice failure modes a fixed taxonomy's authors would have missed.
  • scratch/ — the mutator's ephemeral workbench: provisional hypotheses, half-written tools, and the artifacts from its optional proxy evals (which trigger the inner loop too, but land here, not in rollouts/). Not saved — discarded at the end of the rollout; only what graduates to harness/, tools/, or lessons/ persists.
  • lessons/ — only verified lessons, promoted out of scratch/ once they survive a held-out check (“the citation tool cut errors but added 3 steps”). The durable memory the next mutator reads back.

The mutator works by writing code: it edits the harness (tools, skills, scaffold) and records what it learns in lessons/ — each lesson starting in scratch/ as a hypothesis, promoted to lessons/ only after a held-out check (grounded in eval counts, not two cherry-picked traces). An optional capability we add later lets it also experiment: run its own cheap proxy evals to test an edit before paying for a full rollout, and — since the seeded proxy is right there in tools/ — even fork it into a sharper one (a cheaper agent stand-in or judge) in scratch/ for that private eval, under a compute budget. Those proxy evals stay the mutator's private instrument: it can sharpen what it uses to decide, but it cannot touch the canonical standard eval OpenEvolve selects on (§4) — or it would be grading its own homework.

The research agent and the Harvey agent are two distinct agents — one does legal work, the other writes code to modify its harness. (They happen to be the same kind of object, an agent harness, so the same machinery could later be turned on the research agent's harness too — the TTT direction in §5, not today.) The open empirical question to measure first: does the richer (and pricier) agentic proposer beat a cheap single call per unit of compute? The agentic proposer is the new cost we're adding, and §4's wins all reduce the other cost — so this trade is the experiment that gates whether the agentic proposer is worth its cost.

3 · The inner loop — 0th-order, no gradients

The inner agent is “trained” without touching its weights. We don't RL-finetune GLM-5.2; we change the harness around it — the tools it's handed, the skill in its prompt, what its memory carries. That is the 0th-order move: the research agent changes the inner agent's behavior through the harness, not through gradients. The 0th-order move needs no training infrastructure, so the inner loop ships first, as the cheaper half of the system. For now — the far end of the ladder (§1) is the mutator writing training code that updates these weights, at which point the inner loop leaves 0th-order. Note that's a different escalation from TTT on the proposer (§5): this trains the agent being optimized; TTT trains the optimizer.

Crucially, we push it toward using tools more efficiently, not just scoring higher — Harvey's own data shows the trained agent cut reads 44→24. So the reward is quality and cost (steps / tokens), and efficiency also becomes an archive axis (§5) — otherwise a tool that doubles pass-rate by tripling tool calls would look like progress.

The inner loop is deliberately simple: take the improved harness and run it on the Harvey LAB dataset (our calibration set) — one pass over the task set — producing a rollout (trace + deliverable) per task. There is no per-rollout sandbox or special machinery here; the shell, code, and filesystem all live in the outer loop (the mutator's sandbox — PR #86 + PR #87, §2), not here. This canonical run is the standard eval's (§4); its trajectories are dumped into the outer loop's rollouts/ and carried over for the mutator to read next round. (The mutator's optional proxy evals also run the inner loop, but their artifacts land in scratch/ — ephemeral, not carried.) Today it's a single pass per task; a within-task draft→check→revise loop is a later option.

4 · The reward — and why its cost gates the search

Two evals run here, and keeping them apart matters. While it researches, the mutator fires its own proxy evals (§2) — cheap, private, just to decide what to propose next (optionally, later, it may even build its own sharper proxies for this). But the filesystem is OpenEvolve's state, and to choose which state to mutate next OpenEvolve needs a canonical score. So once the research agent produces an improved harness, a standard eval runs at the end of the rollout. It scores that harness on the full calibration set, graded by the trusted ensemble judge, into one combined_score. That number — not the mutator's private proxy evals — is what OpenEvolve files into the MAP-Elites archive and selects on. The rest of this section is about making that eval cheap, because it runs every step.

Scoring one candidate means running the harness on N calibration tasks and judging every deliverable. With a strong agent and a strong judge, this dominates cost — so the search is only as affordable as the reward is cheap. Three validated results from prior work make it affordable, each cutting a different factor of N × (agent + judge):

① Ensemble judge validated

gpt-5-mini + gpt-5, escalate to Sonnet only when they disagree (~1.86% of criteria — that's the ~50× Sonnet-call reduction). 0.16% false-pass. Grading stops being the bottleneck.

② Revise proxy validated

Predict a skill's lift without re-running the agent — revise the bare deliverable with the skill in one call. in-set r=0.76 (measured on the calibration set; out-of-set not yet validated), ~3.6× cheaper. A cheap fidelity rung.

③ GLM-5.2 endpoint validated

The agent itself: GLM-5.2 beats Sonnet on the harness (0.843 vs 0.800 on our 30-task headroom set; 19 wins / 7 ties / 4 losses) at ~3× lower cost. Cheaper rollouts.

Plus a fidelity cascade (smoke→proxy→full) and engineered eval sets (stratify / discriminate / rotate) so budget is spent only where candidates separate. Full results: the cost-proxy site.

Mind the noise floor. The reward is stochastic — our own smoke evals showed an A+ harness swinging ~6 points run-to-run. Evolving on a signal below that floor just promotes lucky harnesses. So a candidate has to beat its parent by more than the measured run-to-run σ (more seeds / tasks, not one pass) before it's archived, and the cascade's rungs are sized by variance, not only by cost.

Keeping the proposer honest. The proposer's job is literally to maximize an LLM judge — a textbook reward-hacking setup, and worse here because the mutator can read the judge's behavior in the dumped traces and even launch its own evals. So the gates have to be load-bearing, not a list: (1) a held-out judge the search never sees — and when the search judge passes but the held-out judge fails above a small rate, freeze that lineage and audit; that divergence is the reward-hacking alarm. (2) A programmatic grounding check (string-match every cited span against the source, not another LLM that citation-shaped strings can fool). (3) Withhold the judge's per-criterion rationale from the dump so the mutator can't optimize the judge directly. (4) A small human-audited canary set. The final verdict is always the most trusted (Sonnet/ensemble) judge on held-out tasks — a weak search judge costs time, never a wrong conclusion.

5 · The hard parts (design choices & open questions)

The two most novel mechanisms — the agentic proposer and self-generated tools — are also the least built. Here is where the real design decisions live.

What can the mutator change? (the edit ladder)
Because every edit is code, the editable surface widens as we grant more power: skill (prose) → tools (new actions) → scaffold (compaction, planner, validate→revise) → eventually training code that updates the Harvey agent's weights. We climb it lowest-risk-first; each rung needs stronger sandboxing and validation than the last, and the top rung (touching weights) stays gated behind the cheaper rungs actually paying off. The same power could later let it improve its own instruments (an optional add-on, §2): a cheaper agent stand-in or sharper judge proxy for its private eval — never the canonical standard eval it's scored by. That code-can-do-anything power is why the sandbox and budget guards (below) are load-bearing, not nice-to-have.
What is a “new tool,” and how is it born?
Concretely: the proposer writes a small Python function with a typed signature (e.g. a citation-checker that verifies every cited span exists in the source). It must pass a validation rung before it can score — does it run, does it improve a held-out micro-task (including adversarial cases where the right answer is “fail”, so a tool that always returns true is caught) — all inside the mutator's sandbox (§2), isolated from the host. Once it passes, it graduates from scratch/ into the harness (harness/) and the Harvey agent can call it. A “skill” (prose) edits the prompt; a “tool” (code) extends the Harvey agent's action space — both are edits to harness/. (A proxy the mutator builds for its own evals is a different thing: that lives in tools/, §4.) We start with skills (lower risk) and graduate to tools.
How do tools live in the archive?
A harness variant's genotype is its persisted filesystem state — harness/ + tools/ + the carried lessons/ + the dumped rollouts/ (plus the read-only knowledge/); the ephemeral scratch/ is dropped. Tools are namespaced and content-hashed so children inherit them without collision, and a tool-bloat guard (a cost axis + periodic consolidation) retires redundant harness tools — otherwise “build on winners” accretes cruft and inflates the Harvey agent's prompt until more tools make it worse.
What are the MAP-Elites axes?
MAP-Elites is only as good as its behavior axes, and most of the obvious ones — skill-file length, number of tools, steps/task, cost/task — are resource axes (the efficiency signals from §3). The grid needs at least one capability axis too, or it only preserves variety in how expensive a harness is, not in what it's good at: e.g. a per-task-cluster performance bucket (drafting vs. retrieval vs. cite-checking). The natural “where does it fail” axis can't come from the feedback.py taxonomy we're replacing — it's derived instead from the mutator's own lessons/ (§2). Together these keep the search from collapsing to a global top-k.
Search vs. evaluation budget.
One outer proposal fans out to N×k inner rollouts; the agentic proposer is itself now multi-step and pricey. The cascade promotes a candidate smoke→proxy→full so most die cheap, and the proposer's per-edit cost is tracked against the single-call baseline — that ratio is the core experiment (§2).
Won't dumping everything blow the mutator's context?
The dump is large — the N×k traces plus the full harness. The mutator doesn't read it all into context; it gets a filesystem + shell and navigates it (grep for the failures, diff the harness against its parent, sample a few traces, re-run one case), the way an engineer triages a log directory. So “dump everything” is cheap to produce (just write the inner loop's artifacts to disk) and the mutator pays only for the slice it chooses to read.
Evolution now, TTT later.
Today the mutator is frozen (pure evolution — improvement accumulates as scored harness variants in the archive and verified findings in lessons/). The speculative next step, and the repo's namesake, is test-time training: RL-train the proposer so it specializes at proposing harness edits (distinct from the §1 rung where the mutator trains the inner agent — here we train the proposer). It isn't free — training it on the same reward the archive is selected on couples two optimizers to one signal (compounding the reward-hacking risk above), and the target is non-stationary (the harness it edits keeps changing). So TTT likely needs a held-out reward channel distinct from the search judge, or it just amplifies whatever that judge is blind to. Same loop, consumed twice — carefully.

7 · Where we are, and the gap

Today (the harvey-evolve-prototype branch): a complete, well-engineered single-loop optimizer. OpenEvolve mutates the skill-file prose, injects it via HARVEY_PLAYBOOK, runs the Harvey harness, scores with the ensemble judge, files variants into MAP-Elites, and feeds a heuristic failure-mode taxonomy (its feedback.py) back into the next mutation. The reward and the cost discipline are production-quality. But the “proposer” is one LLM rewrite, the feedback is a fixed heuristic, the artifact is a single markdown file, and there's no inner agent.

Landing now: the inner-loop substrate — PR #86 (filesystem) and PR #87 (local shell) — plus PR #88 wiring OpenEvolve to NanoRollout (on a sibling branch). These turn the single prose-rewrite into a real agentic loop.

The gap to close (this design):

  • An inner 0th-order agent with the filesystem + shell that iterates within a task (draft→check→revise) and reads the logged trajectories — not one rewrite per outer step.
  • A genuinely agentic proposer (warm-started + tool-using), not just a seeded artifact mutated by a bare LLM.
  • Raw-dump feedback, not a taxonomy. Replace feedback.py with dumping the inner loop's whole evidence package (all trajectories + the harness code) to the filesystem, and let the agentic mutator extract its own feedback and propose the next tools & skills (§2).
  • A verify → new-tools path (§5) that grows the action space — synthesize, validate, and register tools — rather than only rephrasing a playbook, kept honest by anti-gaming gates.

The target end state: a research agent that writes the Harvey agent's tools, validates them on cheap rollouts, and keeps the ones that survive on held-out tasks — optimizing the Harvey harness automatically, with no gradient step in the loop.

8 · A later direction: ToolOrchestra not in v1

A wishful extension we've wanted from the start — flagged here as a separate, post-v1 direction. ToolOrchestra trains a small orchestrator model (8B) to coordinate calls to bigger models and specialized tools, with RL on outcome-, efficiency-, and preference-aware rewards. Instead of one big model running every step, a cheap orchestrator decides per step whether to answer itself or call out — paying for a big model only when a step needs it. The payoff is steep: their 8B orchestrator beats GPT-5 on Humanity's Last Exam (37.1% vs 35.1%) at 2.5× the efficiency, and surpasses GPT-5 on tau2-Bench / FRAMES at ~30% of the cost.

One model everywhere → a cheap orchestrator over a toolbox. Big-model quality on the hard steps, a fraction of the cost on the easy ones.

Two places this drops into the present design — in both, “the model endpoint” simply becomes a richer object:

① The mutator becomes the orchestrator

Make the outer-loop mutator (§2) a small model, TTT-trained (the repo's namesake) as a ToolOrchestra, with the big API models (GPT-5, Claude, Opus) sitting in its tools/. It learns to call the expensive models sparingly. This directly answers §2's open question — does the agentic proposer beat a cheap single call per unit of compute? — ToolOrchestra makes the proposer cheap by construction.

② The Harvey endpoint becomes an orchestra

Today the harness endpoint is one model (GLM-5.2). Because the endpoint lives in harness/, the research agent can rewrite it into a ToolOrchestra — a small orchestrator that routes each step to the right model or legal tool (retrieve, cite-check, draft). Big-model quality on the hard steps, a fraction of the cost elsewhere; just another rung on the edit ladder (§1).

Why it's a clean fit: our reward is already quality and cost (§3–4) — exactly ToolOrchestra's multi-objective signal; the archive already selects on combined_score; and TTT on the proposer (§5) is the very training loop it needs. But it's explicitly out of scope for v1 — it lands only once the base loop works.
