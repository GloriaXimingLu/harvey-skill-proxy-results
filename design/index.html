A harness that improves itself: agentic AlphaEvolve for the Harvey legal agent

An evolutionary search that grows the agent's own tools and skills. A warm-started proposer (GLM-5.2 / Claude) reads how the agent fails and writes a new tool; a 0th-order inner agent (GLM-5.2, weights frozen — only the harness around it changes) uses it to do legal work and produce rollouts (a full run: the trace + the deliverable); those rollouts are dumped to a filesystem, and the proposer reads them to grow the next tool. No gradient step in sight — the harness is the only thing that changes, and it gets better while you sleep.

A legal agent is only as good as its harness — the skill file, the tool set, and the control-flow scaffold wrapped around the model. Today humans hand-tune those. This system makes the harness tune itself. The engine is AlphaEvolve (DeepMind) — an evolutionary search that keeps a population of candidate programs, scores each, and repeatedly mutates the winners. Here the “program” being evolved is the agent's own harness.

In one line: vanilla AlphaEvolve mutates code with a single blind LLM diff and keeps what scores higher. We change two things: (1) the program being evolved is the agent's harness, and (2) — the idea that matters — we replace the blind diff with an agent that investigates: it reads the failing rollouts, diagnoses the cause, and writes a new tool. Everything else (archive, score, evolve the winners) is standard AlphaEvolve.

Two coupled loops drive it: a slow, smart outer loop (one expensive, agentic proposal + verification per step) that edits the harness, and a fast inner loop (many cheap rollouts per proposal) that runs the harness to produce the evidence the proposer learns from.

The whole system in one picture

Five stations, one cycle ①→⑤. ① the archive samples a parent harness → ② the agentic mutator edits it → ③ a 0th-order GLM-5.2 agent runs it (rollouts) → ④ a standard eval scores it (the score OpenEvolve selects on) → ⑤ the inner loop dumps its artifacts — all trajectories + the harness code — into the mutator's filesystem workspace (rollouts/), which it navigates with shell + code alongside its seeded knowledge/ and its own lessons/ to propose the next tools & skills; the scored harness is filed back. Solid = exists/merged, dashed = PR-open or to-build — the PR #86 filesystem exists as a PR; using it as the feedback dump is the new wiring.

Reading the diagram — one turn of the loop

  • Sample a parent harness. The archive — a population of harness variants kept in a diversity grid (MAP-Elites + islands: keep the best variant in each niche, not just one global best) — hands a promising harness (its skill file + tool set) to the outer loop.
  • Propose an edit, agentically. The proposer (GLM-5.2 or Claude), warm-started with Harvey's playbook and our own findings, inspects the parent's failing rollouts, diagnoses a missing capability, and writes a new tool or skill. This is the one change from vanilla AlphaEvolve: an agent that looks — reads traces, can run experiments — instead of a one-shot diff. Applying the edit yields a candidate harness.
  • Run it — 0th-order rollouts. A GLM-5.2 agent runs the candidate harness on the calibration tasks inside a filesystem + local shell, using the proposed tools to do the legal work. “0th-order” = weights frozen, no training run; the agent's behavior changes only because the harness changed. The target is that it iterates within a task (draft → self-check → revise); today it's a single pass. Every step is logged into a rollout (trace + deliverable).
  • Score it — the standard eval. After the mutator finishes its work, the improved harness is graded on the calibration set into one combined_score — success plus cost/steps (§3), so efficiency counts. This is the canonical number OpenEvolve selects on (distinct from the mutator's own mid-research proxy evals), the expensive step, and the focus of months of prior work (§4).
  • Dump everything → the mutator extracts its own feedback. The inner loop dumps its artifacts to the filesystem — every trajectory (trace, deliverable, score) plus the harness code itself. The agentic mutator reads that raw dump, extracts its own feedback (which tool was botched, where steps were wasted, what capability is missing), confirms the edit actually helped, and proposes the next tools and skills. No heuristic taxonomy in between — the mutator decides what matters. The scored harness is filed back into the archive, and the population gets better.

1 · What we optimize: the harness, as code

The harness decomposes into editable regions. The mutator edits them as code, so in principle nothing is off-limits — the ladder below is really about how much power we hand it, lowest-risk first. We start with the skill file (cheapest, highest-variance), then tools, then the control-flow scaffold (e.g. the LAB blog's finding that compaction helps — the mutator can just write it into the loop), and eventually the agent's own weights. Which levers are worth trying first is itself seeded from human knowledge into knowledge/.

Seen this way, improving the harness is a code-optimization problem — an editable artifact plus a numeric reward — exactly what AlphaEvolve needs. Concretely the artifact is a filesystem state (the harness code plus the mutator's knowledge/ and lessons/): that whole directory is one OpenEvolve Program, and it's what the archive stores, mutates, and builds winners on.

2 · The outer loop — an agentic, warm-started proposer

This is the new idea, so it gets the most attention. In vanilla AlphaEvolve a frozen model emits one diff, blind. Here, proposing a better harness is itself an agent task — we call this agent the proposer (a.k.a. the mutator); they are the same object throughout this doc.

Why agentic, not a single call

A good edit can't be guessed blind. It requires looking: read the failing traces, see which tool the agent reached for and botched, try a variant in the shell. So one proposal becomes one agent trajectory — a diagnosis grounded in observed failure, not a shot in the dark.

Why warm-started

The proposer doesn't start from zero. We seed it with everything we already know, dropped on its filesystem as knowledge/: Harvey's LAB blogs (emit an artifact, cite spans, validate→revise, retrieve narrowly) and our own findings — the validated A+ playbook, the GLM-5.2 endpoint, the ensemble judge, and the cost-proxy results (what predicts a skill's lift). So it proposes informed edits from turn one — without parroting any of it (a seeded idea still has to beat the parent on held-out tasks to survive).

The mutator's workspace: one filesystem, structured

The mutator isn't handed a digest — it's handed a filesystem and a shell, and it works like a research engineer. The filesystem (PR #86) is organized into subfolders it reads and writes:

  • knowledge/ — warm-start context, seeded up front (above): the Harvey blogs, our own findings (A+ playbook, GLM-5.2 endpoint, ensemble judge, cost-proxy), and a starter menu of harness dimensions worth trying (compaction, planner, retrieval, memory). The mutator reads what's relevant; it isn't bound by any of it.
  • rollouts/ — the inner loop dumps its artifacts here each round: every trajectory (trace · deliverable · score) plus the harness code itself. The raw evidence — no taxonomy pre-digesting it into fixed buckets (a deliberate reversal of the prototype's feedback.py). A fixed taxonomy only surfaces the failure modes its authors anticipated; the raw dump lets the agent notice the ones they didn't.
  • scratch/ — the mutator's draft space: provisional hypotheses, half-written tools, intermediate eval outputs. Nothing here is trusted yet — it's the lab bench, not the notebook.
  • lessons/ — only verified lessons, promoted out of scratch/ once they survive a held-out check (“the citation tool cut errors but added 3 steps”). This is the durable memory the next mutator reads back — insight accumulates here across iterations, not only as scores in the archive.

Because it has shell + code tools, the mutator doesn't only propose — it experiments: it writes a candidate tool or skill as code, and it can launch its own cheap proxy eval (the cost-proxy of §4) to test whether the edit helps before paying for a full rollout. So the cost-proxy stops being just our offline ranking tool and becomes an instrument the mutator invokes. And because it works by writing code, it isn't limited to our proxy: it can build a better one — a cheaper agent stand-in, a sharper judge proxy — that surpasses our hand-built baselines (seeded in knowledge/) for its immediate eval, all under a compute budget it can't exceed. These proxy evals are the mutator's private instrument, though: it can sharpen what it uses to decide, but it cannot touch the canonical standard eval that OpenEvolve selects on (§4) — that one stays fixed and trusted, or the mutator would be grading its own homework. And a lesson it writes starts in scratch/ as a hypothesis, promoted to lessons/ only after a held-out check — grounded in eval counts, not two cherry-picked traces.

The optimizer and the optimized are the same kind of object — an agent harness improving an agent harness. Because the proposer is itself a harness, the same loop can eventually be pointed at the proposer's own harness — the system improving the thing that improves it. The open empirical question to measure first: does the richer (and pricier) agentic proposer beat a cheap single call per unit of compute? The agentic proposer is the new cost we're adding, and §4's wins all reduce the other cost — so this trade is the experiment that decides the whole project.

3 · The inner loop — 0th-order, no gradients

The inner agent is “trained” without touching its weights. We don't RL-finetune GLM-5.2; we change the harness around it — the tools it's handed, the skill in its prompt, what its memory carries. That is the 0th-order move: behavior improves through the harness, not through gradients. It needs no training infrastructure, so it ships first, and it's the cheaper half of the system. For now — the far end of the ladder (§1) is the mutator writing training code that updates these weights, at which point the inner loop leaves 0th-order. Note that's a different escalation from TTT on the proposer (§5): this trains the agent being optimized; TTT trains the optimizer.

Crucially, we push it toward using tools more efficiently, not just scoring higher — Harvey's own data shows the trained agent cut reads 44→24. So the reward is quality and cost (steps / tokens), and efficiency also becomes an archive axis (§5) — otherwise a tool that doubles pass-rate by tripling tool calls would look like progress.

To do real work and leave evidence, the inner agent runs each rollout in its own scratch workspace — a local shell (PR #87) and a filesystem snapshot (PR #86) it can read and write — isolated per rollout so one run can't contaminate the next (the 0th-order invariant: behavior changes only because the harness changed, not because state leaked). Each run emits a full rollout — trajectory plus deliverable. After scoring, those artifacts are copied into the mutator's separate workspace under rollouts/ (§2) — that copy is the dump the outer loop reads. (Today this is one pass per candidate; the within-task draft→check→revise loop is the next step.)

4 · The reward — and why it's the whole game

Two evals run here, and keeping them apart matters. While it researches, the mutator fires its own proxy evals (§2) — cheap, private, just to decide what to propose next (it may even build its own sharper agent/judge proxies for this). But the filesystem is OpenEvolve's state, and to choose which state to mutate next OpenEvolve needs a canonical score. So once the research agent has finished and produced an improved harness, a standard eval runs at the end of the rollout: that harness on the full calibration set, graded by the trusted ensemble judge → one combined_score. That number — not the mutator's private proxy evals — is what OpenEvolve files into the MAP-Elites archive and selects on. The rest of this section is about making that eval cheap, because it runs every step.

Scoring one candidate means running the harness on N calibration tasks and judging every deliverable. With a strong agent and a strong judge, this dominates cost — so the search is only as affordable as the reward is cheap. Three validated results from prior work make it affordable, each cutting a different factor of N × (agent + judge):

① Ensemble judge validated

gpt-5-mini + gpt-5, escalate to Sonnet only on disagreement (1.86% of criteria). 0.16% false-pass, ~50× fewer Sonnet calls. Grading stops being the bottleneck.

② Revise proxy validated

Predict a skill's lift without re-running the agent — revise the bare deliverable with the skill in one call. in-set r=0.76, ~3.6× cheaper. A cheap fidelity rung for the search.

③ GLM-5.2 endpoint validated

The agent itself: GLM-5.2 beats Sonnet on the harness (0.843 vs 0.800 on our headroom-30, 19/4 head-to-head) at ~3× lower cost. Cheaper rollouts.

Plus a fidelity cascade (smoke→proxy→full) and engineered eval sets (stratify / discriminate / rotate) so budget is spent only where candidates separate. Full results: the cost-proxy site.

Mind the noise floor. The reward is stochastic — our own smoke evals showed an A+ harness swinging ~6 points run-to-run. Evolving on a signal below that floor just promotes lucky harnesses. So a candidate has to beat its parent by more than the measured run-to-run σ (more seeds / tasks, not one pass) before it's archived, and the cascade's rungs are sized by variance, not only by cost.

Keeping the proposer honest. The proposer's job is literally to maximize an LLM judge — a textbook reward-hacking setup, and worse here because the mutator can read the judge's behavior in the dumped traces and even launch its own evals. So the gates have to be load-bearing, not a list: (1) a held-out judge the search never sees — and when the search judge passes but the held-out judge fails above a small rate, freeze that lineage and audit; that divergence is the reward-hacking alarm. (2) A programmatic grounding check (string-match every cited span against the source, not another LLM that citation-shaped strings can fool). (3) Withhold the judge's per-criterion rationale from the dump so the mutator can't optimize the judge directly. (4) A small human-audited canary set. The final verdict is always the most trusted (Sonnet/ensemble) judge on held-out tasks — a weak search judge costs time, never a wrong conclusion.

5 · The hard parts (design choices & open questions)

The two most novel mechanisms — the agentic proposer and self-generated tools — are also the least built. Here is where the real design decisions live.

What can the mutator change? (the edit ladder)
Because every edit is code, the surface grows with how much power we grant: skill (prose) → tools (new actions) → scaffold (compaction, planner, validate→revise) → eventually training code that updates the agent's weights. We climb it lowest-risk-first; each rung needs stronger sandboxing and validation than the last, and the top rung (touching weights) stays gated behind the cheaper rungs actually paying off. The same power lets it improve its own instruments — building a cheaper agent stand-in or a sharper judge proxy for its immediate eval, beyond our hand-built ones — though never the canonical standard eval it's scored by. The same code-can-do-anything power is why the sandbox and budget guards (below) are load-bearing, not nice-to-have.
What is a “new tool,” and how is it born?
Concretely: the proposer writes a small Python function with a typed signature (e.g. a citation-checker that verifies every cited span exists in the source). It must pass a validation rung before it can score — does it run, does it improve a held-out micro-task (including adversarial cases where the right answer is “fail”, so a tool that always returns true is caught) — and it executes inside the harness's Podman sandbox, not the mutator's bare shell, so self-written code can't touch the host or leak state across rollouts. A “skill” (prose) edits the prompt; a “tool” (code) extends the action space. We start with skills (lower risk) and graduate to tools.
How do tools live in the archive?
A harness variant's genotype is its filesystem state — {skill file, tool set, and the carried-over knowledge/ · lessons/}. Tools are namespaced and content-hashed so children inherit them without collision, and a tool-bloat guard (a cost axis + periodic consolidation) retires redundant tools — otherwise “build on winners” accretes cruft and inflates the prompt until more tools make the agent worse.
What are the MAP-Elites axes?
MAP-Elites is only as good as its behavior axes, and most of the obvious ones — skill-file length, number of tools, steps/task, cost/task — are resource axes (the efficiency signals from §3). The grid needs at least one capability axis too, or it only preserves variety in how expensive a harness is, not in what it's good at: e.g. a per-task-cluster performance bucket (drafting vs. retrieval vs. cite-checking). The natural “where does it fail” axis can't come from the deleted feedback.py taxonomy — it's derived instead from the mutator's own lessons/ (§2). Together these keep the search from collapsing to a global top-k.
Search vs. evaluation budget.
One outer proposal fans out to N×k inner rollouts; the agentic proposer is itself now multi-step and pricey. The cascade promotes a candidate smoke→proxy→full so most die cheap, and the proposer's per-edit cost is tracked against the single-call baseline — that ratio is the core experiment (§2).
Won't dumping everything blow the mutator's context?
The dump is large — many long traces plus the full harness. The mutator doesn't read it all into context; it gets a filesystem + shell and navigates it (grep for the failures, diff the harness against its parent, sample a few traces, re-run one case), the way an engineer triages a log directory. So “dump everything” is cheap to produce (just write the inner loop's artifacts to disk) and the mutator pays only for the slice it chooses to read — which is also why we dump the raw artifacts rather than pre-summarizing them.
Evolution now, TTT later.
Today the proposer is frozen (pure evolution — intelligence accumulates in the archive and in lessons/). The speculative next step, and the repo's namesake, is test-time training: RL-train the proposer so it specializes at proposing harness edits (distinct from the §1 rung where the mutator trains the inner agent — here we train the proposer). It isn't free — training it on the same reward the archive is selected on couples two optimizers to one signal (compounding the reward-hacking risk above), and the target is non-stationary (the harness it edits keeps changing). So TTT likely needs a held-out reward channel distinct from the search judge, or it just amplifies whatever that judge is blind to. Same loop, consumed twice — carefully.

7 · Where we are, and the gap

Today (Thomas's harvey-evolve-prototype): a complete, well-engineered single-loop optimizer. OpenEvolve mutates the skill-file prose, injects it via HARVEY_PLAYBOOK, runs the Harvey harness, scores with the ensemble judge, files variants into MAP-Elites, and feeds a heuristic failure-mode taxonomy (its feedback.py) back into the next mutation. The reward and the cost discipline are production-quality. But the “proposer” is one LLM rewrite, the feedback is a fixed heuristic, the artifact is a single markdown file, and there's no inner agent.

Landing now (Jonas): the inner-loop substrate — PR #86 (filesystem) and PR #87 (local shell) — plus PR #88 wiring OpenEvolve to NanoRollout (on a sibling branch). These turn the single prose-rewrite into a real agentic loop.

The gap to close (this design):

  • An inner 0th-order agent with the filesystem + shell that iterates within a task (draft→check→revise) and reads the logged trajectories — not one rewrite per outer step.
  • A genuinely agentic proposer (warm-started + tool-using), not just a seeded artifact mutated by a bare LLM.
  • Raw-dump feedback, not a taxonomy. Replace feedback.py with dumping the inner loop's whole evidence package (all trajectories + the harness code) to the filesystem, and let the agentic mutator extract its own feedback and propose the next tools & skills (§2).
  • A verify → new-tools path (§5) that grows the action space — synthesize, validate, and register tools — rather than only rephrasing a playbook, kept honest by anti-gaming gates.

The north star: the harness writes its own tools, validates them on cheap rollouts, keeps the ones that survive on held-out tasks — a legal agent that gets better while you sleep, with no gradient step in sight.
